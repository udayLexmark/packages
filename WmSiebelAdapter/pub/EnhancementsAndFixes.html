1.0 Enhancements

webMethods Siebel Adapter 6.0 SP3 includes the following enhancements.

1-11PCEK
The Siebel Adapter services should accept credentials at run time.

webMethods Siebel Adapter allows the configuration of the Siebel connections through the Integration Server console. However, this configuration is static. The user cannot pass the credentials during run time, while executing the adapter.

This issue is resolved by allowing you to dynamically provide the user name and password at run time, to connect to the Siebel server.

1-TITBJ
Case insensitive search is not supported in Siebel Adapter.

While configuring adapter services, when searching for a selected business component, the case insensitive search is not supported.

This issue is resolved by providing an option ‘~=’ for case insensitive search in the Operator field.

2.0 Included Fixes and Service Packs

webMethods Siebel Adapter 6.0 SP3 includes the following fixes and service packs that were shipped since the last release.

1-RD7CB (SBL_6-0_Fix4)
Output fields in services for the Siebel Adapter cannot be renamed after applying Siebel Adapter 6.0 SP2.

This issue is resolved.

1-RJ72A (SBL_6-0_Fix5)
The Query Business Components no longer work.

This is due to changes in error codes for Siebel Server 7.7.1 that caused the Query Business Component to not function.

This issue is resolved.

1-RKWCZ (SBL_6-0_Fix5)
EAI webMethods Transports are not imported into Siebel 7.7.1.

You cannot import Siebel 7.5 SIF files into Siebel 7.7.1 due to schema changes in the release of Siebel 7.7.1.

This issue is resolved.

1-THSRZ (SBL_6-0_Fix6)
EAI webMethods Transports append corrupted characters to the error message field.

This issue is resolved.

1-UNBOD (SBL_6-0_Fix7)
The SyncISInvoke method of EAI webMethods Transports returns the output arguments: documentData, status, and message. However, the first output argument should be outputData, not documentData.

This issue is resolved.

1-SXAKO (SBL_6-0_Fix8)
While loading data into the Siebel system, an error occurs indicating that the current insert is not possible because the data already exists. The following messages appear in the Server Log file:

If you would like to enter a new record, please ensure that the field values are unique. (SBL-DAT-00381)) 2004-10-14 00:00:13 [ART.0114.1007E] Adapter Runtime: Error Logged.

See Error log for details. Error: [ADA.448.1000] SiebelJavaOperations.performInsert() - threw a SiebelException: code(27714), msg(A record that contains identical values to the record you have created already exists.

This error occurs because the Siebel Adapter is inserting the same record twice.

This issue is resolved.

1-TK8PL (SBL_6-0_Fix9)
The Siebel Adapter throws a java.text.ParseException when running a Query service on a boolean field that does not contain a value.

This issue is resolved.

1-YHLH7 (SBL_6-0_Fix12)
The Siebel Adapter Connection Management properties "Startup Retry Count" and "Startup Backoff Timeout (sec)" are not working.

This issue is resolved.

1-UFH45 (SBL_6-0_Fix13)
There is no .access file in the pub directory for the Siebel Adapter.

The Siebel Adapter does not have a .access file in the pub directory or any of the subsidiary directories (for example, doc, doc\OnlineHelp). Without this file, any authenticated user can retrieve any of the files in that directory, not just administrators.

This issue is resolved.

1-RY735 (SBL_6-0_Fix13)
The Siebel Adapter displays an "Invalid Boolean String" error message for boolean fields that contain values other than the Siebel default values.

The Siebel default values for boolean fields are Y or N. If an organization decides to use values other than the defaults allowed by Siebel (for example, T or F), the Siebel Adapter detects these values as being invalid and throws an error. In addition, no manipulation of the "invalid" boolean fields is allowed.

This issue is resolved, and the Siebel Adapter now recognizes the following values as boolean: Y/N, YES/NO, T/F, TRUE/FALSE, 1/0.

1-YXISN (SBL_6-0_Fix14)
When using the EAI webMethods Transports, Siebel Server cannot invoke services on the Integration Server.

This situation occurs when using the Siebel Business Service Simulator to make a call to the Integration Server using an XML document, and causes the simulator to hang when the EAI webMethods Transports methods are invoked. The Siebel Server is using JVM 1.4.2, and the Siebel Server log reports the following error: "ObjMgrBusServiceLog InvokeMethod 4 0 2005-08-04 21:35:16 Begin: Business Service 'EAI webMethods Transports' invoke method: 'SyncISInvoke' at 43b0010". This error occurs because the wMISTransport.so shared object cannot load the JVM.

This issue has been resolved. The wMISTransport code has been updated to include the VMOPTIONS -Xusealtsigs option to ensure that the signal handlers used by the Siebel Server do not conflict with those of the JVM. With this option, the JVM now loads successfully into the application object manager.

1-101NQZ (SBL_6-0_Fix14)
EAI webMethods Transport causes the Siebel Server to crash.

This situation occurs when the user logs off from Windows after invoking an EAI webMethods Transport business service. When the user logs in again, one of the Siebel processes (siebmtshmw.exe) is missing from the Task Manager, preventing the user from logging in to the Siebel application. The Siebel Server log reports the following error:

"SisnTcpIp SisnSockError 1 2005-06-11 19:20:02   1904: [TCPIP- client] recv() failed for sd=1236 (err=10054 | An existing connection was forcibly closed by the remote host.)"

This issue has been resolved. The wmISTransport.dll has been updated to include the -Xrs option which is used when invoking the JVM from the DLL. Now, after logging off from Windows, the Siebel process (siebmtshmw.exe) continues to run.

1-VMOVB (SBL_6-0_Fix14)
The client software hangs after invoking the getTxData method when the Integration Server is down.

Calling the EAI webMethods Transport getTxData method when the Integration Server is down causes the Siebel Server to hang. This issue is caused by an error in an Integration server component (client.jar).

This issue is resolved.

1-10HSXR (SBL_6-0_Fix14)
Need error handling when using AsyncISInvokeG and an incorrect 'server name' is given.

There is no mechanism for handling the error that occurs when the Integration Server passes incomplete or incorrect parameters back to the EAI webMethods Transport AsyncISInvokeG business service. The GetTxData method returns the status as reported by the Integration Server, but it does not return the errors that may have happened while reporting the status back to Siebel. Those errors are logged into the Transport/ErrorLogs folder.

This issue has been resolved. The getTxData method has been updated to retrieve the contents of the .err file and concatenate it with the status reported by the Integration Server and return it to the caller.

1-12CISR (SBL_6-0_Fix15)
The Siebel Adapter executes a wildcard pick when inserting data into Siebel fields that have pick lists, and does not throw an error.

If the value provided for the pick list field is not contained in the pick list, the Siebel Adapter incorrectly picks a value that is similar to the value provided from the pick list. The Siebel Adapter does not throw an error at this time to alert the user.

This issue is resolved. The Siebel Adapter now picks the value only if the exact value provided exists in the pick list. If the value provided for the pick list field does not exist in the pick list, the adapter aborts the insert operation and issues a java exception.
